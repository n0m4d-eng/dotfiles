# Dock / external monitor hotplug fix

Workaround for a Razer Blade Stealth 13 (2019) hardware bug where the external
monitor (via USB-C dock) only comes up reliably right after a reboot. Affects
niri on Ubuntu, i915 graphics.

## Root cause

Two compounding issues, both traced back to the laptop's BIOS (1.02, 2019 —
no fwupd update available for this model):

1. **Hotplug**: after connecting the dock, the Thunderbolt/USB-C controller
   only negotiates **2 DP lanes** (split with USB 3.x) instead of the 4 lanes
   the BIOS sets up at boot. Max usable mode after hotplug is **4K@30**
   (or 1440p@60) — never 4K@60.
2. **Disconnect corruption**: on unplug, i915 logs
   `*ERROR* Timeout waiting for FEC live state to get disabled` because the
   AUX channel disappears before i915 finishes disabling FEC. This corrupts
   i915's DP port state, so the *next* hotplug fails silently (niri sees
   DP-1 connected and sets a mode, but no signal reaches the monitor).
3. **Suspend/resume** (separate case): waking from suspend with the dock
   attached leaves the monitor blank even though i915 reports DP-1 as fully
   connected/enabled with zero errors. The EC's USB-C Alt-Mode connection
   manager only re-negotiates DP lanes on a physical cable hotplug
   (CC-line debounce) — never on resume — and Linux has no ACPI/UCSI handle
   to poke it (`USBC000:00._STA` reports 0/"not present"). There is no
   software fix for this case; see Mitigation below.

## What's in this folder

| Path | Purpose |
|---|---|
| `bin/dock-hotplug.sh` | Runs on dock connect. Forces an i915 DP connector re-detect, then trains the link at 1080p before stepping up to 4K@30 (jumping straight to 4K sometimes fails to negotiate). |
| `bin/dock-disconnect.sh` | Runs on dock disconnect. Unbinds/rebinds the Thunderbolt PCI controller (`0000:00:0d.2`) so it reinitialises cleanly for the next connect. **Currently unreliable** — the udev `remove` rule doesn't always fire (USB remove events don't reliably carry `ID_VENDOR_ID`/`ID_MODEL_ID` env vars for ATTR matching). Kept in case udev matching is fixed later; the real recovery path is `monitor-recover.sh`. |
| `bin/monitor-recover.sh` | Manual recovery: turns DP-1 off, forces a sysfs connector re-detect, turns DP-1 back on. Fixes a blank monitor without a reboot. Bound to `Mod+F10`. |
| `udev/99-dock-hotplug.rules` | Fires `dock-hotplug.sh` when the dock's USB ethernet chip (Realtek `0bda:8153`) appears. |
| `sudoers.d/monitor-recover` | Lets `monitor-recover.sh` run passwordless via sudo (needed for `niri msg` as the target user + sysfs writes as root). |
| `niri/binds-snippet.kdl` | `Mod+F9` (disable DP-1 before unplugging) and `Mod+F10` (run recovery script) keybindings. |
| `systemd/swayidle.service` | Idle daemon config with the suspend/resume mitigation baked in (see below). |
| `install.sh` | Installs the scripts, udev rule, and sudoers rule system-wide. Run with `sudo`. |

## Install

```bash
cd dock-hotplug-fix
sudo ./install.sh
```

Then finish the manual steps it prints:

1. Merge `niri/binds-snippet.kdl` into your niri config (e.g. `~/.config/niri/config.kdl` or an included file).
2. Copy `systemd/swayidle.service` to `~/.config/systemd/user/swayidle.service` if you want the suspend/resume mitigation, then:
   ```bash
   systemctl --user daemon-reload
   systemctl --user restart swayidle
   ```

**Note:** the scripts hardcode the username `n0m4d` and UID `1000` (for
`runuser -u` and locating `/run/user/1000/niri.*.sock`). Adjust if deploying
under a different user.

## Usage

- **Before unplugging the dock**: press `Mod+F9` (or the hotplug will likely
  corrupt DP port state and the next connect will fail silently).
- **On connect**: nothing needed — `dock-hotplug.sh` runs automatically via
  udev and brings the monitor up at 4K@30 after ~11 seconds. Watch progress
  with `journalctl -f -t dock-hotplug`.
- **If the monitor comes up blank anyway** (missed `Mod+F9`, or after a
  suspend/resume with the dock attached): press `Mod+F10`.

## Persistent display arrangement

Once the arrangement (position/scale/mode) is correct, it's saved in
`~/.config/niri/dms/outputs.kdl` (generated by DankMaterialShell, included
from `config.kdl`) so it survives reboots. Since the dock only ever
negotiates 2 lanes on hotplug, the configured mode for the external monitor
should be `3840x2160@30.000`, not the 60Hz preferred mode — if it's set to
60Hz, niri will fail to apply it after a hotplug and may fall back to a
different (wrong) arrangement.

## Mitigation for suspend/resume (no real fix exists)

`swayidle.service` gates the idle-timeout auto-suspend action on `DP-1` NOT
being connected — i.e. auto-suspend via idle timeout only fires when
undocked. While docked, hitting the idle timeout just locks + DPMS-blanks
the screen (no `systemctl suspend`), sidestepping the bug entirely. This was
chosen over always rebooting/replugging after a manual suspend.

## Remaining real fix

A BIOS update from Razer (needs Windows or an EFI USB) would likely fix the
Alt-Mode re-negotiation bug at the root, for both the hotplug and
suspend/resume cases. Untested alternative: `i915.enable_dp_mst=0` in GRUB
might change the DP teardown path and avoid the FEC timeout — worth trying
before chasing a BIOS update.
